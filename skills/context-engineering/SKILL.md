# Context Engineering — 上下文工程

管理 AI Agent 看到什么、何时看到、如何组织。上下文是影响 Agent 输出质量的最大杠杆。

## 上下文层级

从最持久到最临时，分 5 层管理：

```
┌─────────────────────────────────────┐
│ L1: Rules Files (CLAUDE.md 等)      │ ← 始终加载，项目级
├─────────────────────────────────────┤
│ L2: Spec / 架构文档                  │ ← 按功能/会话加载
├─────────────────────────────────────┤
│ L3: 相关源代码文件                    │ ← 按任务加载
├─────────────────────────────────────┤
│ L4: 错误输出 / 测试结果               │ ← 按迭代加载
├─────────────────────────────────────┤
│ L5: 对话历史                         │ ← 持续累积，自动压缩
└─────────────────────────────────────┘
```

## L1: Rules Files（项目规则）

**最高杠杆的上下文。** 始终加载，定义 Agent 在项目中的行为规范。

### 必须包含的内容

```markdown
# PROJECT_RULES.md

## 技术栈
- 框架、语言版本、关键依赖

## 编码规范
- 命名约定（文件/变量/组件）
- 文件组织结构
- 导入顺序

## 禁止事项
- ❌ 不要做的事

## 常用命令
- 构建、测试、部署命令

## 关键路径
- 重要目录说明
- 配置文件位置
```

### 优化原则

- **精炼** — 每条规则必须直接影响输出质量，删掉废话
- **具体** — "使用 pnpm" > "使用包管理器"
- **示例** — 给对/错示例比解释规则更有效
- **维护** — 规则过时比没有规则更糟糕

## L2: Spec / 架构文档

**功能级上下文。** 在开始一个功能时加载，完成后卸载。

- 当前功能的 Spec（来自 spec-driven skill）
- 相关的架构决策记录（ADR）
- 涉及的 API 文档

## L3: 源代码文件

**任务级上下文。** 精准加载，避免信息过载。

### 文件选择策略

1. **先加载入口文件** — 理解整体结构
2. **再加载变更相关文件** — 理解修改范围
3. **最后加载依赖文件** — 理解上下文约束

### 文件加载上限

- 单次任务最多加载 8-10 个文件
- 超过时优先加载接口/类型定义，跳过实现细节
- 用 `grep` 定位关键行，不要加载整个大文件

## L4: 错误输出 / 测试结果

**迭代级上下文。** 每次修改后更新。

- 编译错误 → 只粘贴相关错误信息
- 测试失败 → 包含失败测试的输出
- 运行时错误 → 包含堆栈跟踪

### 优化原则

- **不要粘贴整个构建日志** — 只粘贴错误部分
- **标记哪些是新的** — "这次新增的错误" vs "之前的错误"
- **说明已尝试的修复** — 避免重复建议

## L5: 对话历史

**自动累积，需要主动管理。**

### 何时清理上下文

- 切换到完全不同的任务时
- Agent 开始重复之前的错误时
- 对话超过 20 轮且方向偏移时

### 压缩策略

- 总结前面的决策和结果
- 只保留最终状态，丢弃中间过程
- "我们之前决定用 X 而不是 Y 因为 Z"

## 输出质量下降的诊断

| 症状 | 原因 | 解决方案 |
|------|------|---------|
| Agent 幻觉 API | 缺少 L1 规则文件 | 补充技术栈和 API 约束 |
| 忽略项目规范 | L1 过时或未加载 | 检查 rules file 是否完整 |
| 修改不相关的文件 | L3 加载过多 | 精简加载的文件列表 |
| 重复犯同一错误 | L4 上下文未更新 | 粘贴最新错误信息 |
| 越来越慢/偏离主题 | L5 历史过长 | 开启新会话或压缩上下文 |
| 不遵循指令 | L1 规则太模糊 | 添加具体示例和禁止事项 |

## 与其他 Skill 的关系

- `spec-driven` 生成的 Spec 就是 L2 上下文
- `code-reviewer` 需要加载变更文件的 L3 上下文
- `idea-refine` 输出的方案简报可作为 L2 架构文档

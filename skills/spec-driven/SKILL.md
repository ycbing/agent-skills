# Spec-Driven Development — 规格驱动开发

先写 Spec 再写代码。Spec 是你和工程师之间的共同真理源。

## 何时使用

- ✅ 启动新项目或新功能
- ✅ 需求模糊或不完整
- ✅ 改动涉及多个文件或模块
- ✅ 即将做架构决策
- ✅ 任务预计超过 30 分钟

- ❌ 单行修复、typo、需求明确的微小改动

## 门控流程

```
SPECIFY ──→ PLAN ──→ TASKS ──→ IMPLEMENT
   │          │        │          │
   ▼          ▼        ▼          ▼
 人工审核   人工审核   人工审核   人工审核
```

**不通过当前门不得进入下一阶段。**

---

### Phase 1: SPECIFY（规格定义）

**输入**: 模糊的需求描述

**输出**: 结构化 Spec 文档，包含：

```markdown
# Spec: [功能名]

## 背景
（为什么要做这个？现状是什么？）

## 目标
（做完后期望达到什么效果？1-3 条）

## 非目标
（明确不做哪些事，防止范围蔓延）

## 用户故事
作为 [角色]，我希望 [功能]，以便 [价值]

## 功能需求
1. [具体功能描述]
2. ...
3. ...

## 技术约束
- 框架/语言：...
- API 依赖：...
- 性能要求：...

## 验收标准
- [ ] 标准 1
- [ ] 标准 2
- [ ] ...

## 开放问题
- 问题 1 → 待讨论
- 问题 2 → 待讨论
```

**人工审核点**: 用户确认 Spec 内容准确、完整。

---

### Phase 2: PLAN（方案规划）

**输入**: 已确认的 Spec

**输出**: 技术方案，包含：

```markdown
# Plan: [功能名]

## 架构概览
（关键模块和数据流）

## 技术选型
| 需求 | 选择 | 理由 |
|------|------|------|

## 数据模型
（新增/修改的表/字段）

## API 设计
| 方法 | 路径 | 说明 |
|------|------|------|

## 关键文件清单
| 文件 | 操作 | 说明 |
|------|------|------|

## 风险评估
1. 风险 → 缓解措施
```

**人工审核点**: 用户确认技术方案合理。

---

### Phase 3: TASKS（任务拆解）

**输入**: 已确认的技术方案

**输出**: 按顺序排列的实施任务列表：

```markdown
# Tasks: [功能名]

## Phase 1: 基础设施
- [ ] 任务 1（预计 X 分钟）
- [ ] 任务 2

## Phase 2: 核心功能
- [ ] 任务 3
- [ ] 任务 4

## Phase 3: 完善
- [ ] 任务 5
- [ ] 任务 6
```

**规则**:
- 每个任务预计不超过 30 分钟
- 任务间有明确依赖关系
- 标注优先级（P0/P1/P2）

**人工审核点**: 用户确认任务拆解合理，开始实施。

---

### Phase 4: IMPLEMENT（实施）

**输入**: 已确认的任务列表

**输出**: 逐任务执行代码，每完成一个任务报告进度。

**规则**:
- 严格按任务顺序执行
- 每完成一个任务确认后再继续
- 遇到阻塞及时报告，不自行绕过
- 如果发现 Spec 需要修改，回到 Phase 1 重新确认

---

## 使用方式

直接说"先规划一下"或"写个 spec"，然后描述你的需求。我会引导你走完整个流程。

## 与 idea-refine 的关系

- `idea-refine` 用于**想法模糊**时的发散收敛
- `spec-driven` 用于**想法清晰**但需要结构化实施时的门控流程

可以先 idea-refine 打磨方向，再 spec-driven 规划实施。
